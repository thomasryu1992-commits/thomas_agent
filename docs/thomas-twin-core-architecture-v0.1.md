# Thomas Twin Core Architecture v0.1

Status: Draft  
Purpose: Common architecture and I/O contracts before implementation  
Primary interface: Telegram  
Runtime target: Server-hosted autonomous agent organization

Architecture Note: The official organization architecture is now maintained in [Thomas Autonomous Organization Architecture v0.1](./thomas-autonomous-organization-architecture-v0.1.md). This document remains useful for shared runtime components and I/O contracts.

Operating Note: Runtime behavior, Telegram control, approval, memory promotion, execution limits, and learning conditions are governed by [MVP Operating Policy v0.1.1](./MVP_OPERATING_POLICY.md).

## 1. Document Position

This document fixes the shared backbone of Thomas Twin before deciding the final department structure.

Department names such as Communication Thomas, Research Thomas, Memory Thomas, and Content Thomas are examples only. They are useful for early testing, but the system must not depend on a fixed department list.

The stable design target is the operating structure below:

```text
Real Thomas
  -> Telegram Gateway
  -> Thomas Prime
  -> Department Router
  -> Department Thomas
  -> Agent Runtime / Program Runtime
  -> Validation Layer
  -> Policy and Permission Engine
  -> Restricted Execution Service
  -> Memory System
  -> Audit Log
  -> Telegram Response
```

## 2. Core Principles

1. Thomas Prime owns the common identity, values, long-term goals, decision model, and risk boundaries.
2. Departments are replaceable organizational units, not hard-coded foundations.
3. Agents handle judgment, interpretation, planning, option generation, and complex review.
4. Rule-based Programs handle repeatable, deterministic, testable processing.
5. Validation is independent from creation whenever risk or ambiguity is meaningful.
6. Agents do not directly perform external actions.
7. External actions are executed only by Restricted Execution Service after policy checks.
8. Telegram is the main human interface for requests, approvals, status updates, and final responses.
9. Memory is split into shared Family Memory and specialized Department Memory.
10. Every meaningful decision, execution, and memory update must be auditable.

## 3. Runtime Components

### 3.1 Telegram Gateway

Receives requests from Thomas and sends back questions, approval requests, progress updates, and final results.

Responsibilities:

- Receive Telegram messages.
- Normalize user input into a Task.
- Preserve Telegram metadata for traceability.
- Send approval prompts generated by the Policy Engine.
- Send final and intermediate responses.
- Avoid making strategic decisions by itself.

### 3.2 Thomas Prime

The top-level coordinator and guardian of Thomas DNA.

Responsibilities:

- Interpret the intent of a Task.
- Apply Thomas Identity, Values, Goals, Decision Model, and Risk Constitution.
- Route work to a department or directly to a program when appropriate.
- Decide whether a new department proposal is needed.
- Report important conflicts, risks, and exceptions to Thomas.

### 3.3 Department Router

Selects the best department candidate for a Task.

Responsibilities:

- Match task intent to available departments.
- Detect when no department is suitable.
- Support temporary or shadow departments without changing core architecture.
- Record routing reasons.

### 3.4 Department Thomas

The responsible coordinator for a domain.

Responsibilities:

- Manage department goals and priorities.
- Choose Agents and Programs.
- Integrate results from multiple roles.
- Resolve internal conflicts.
- Report important outcomes to Thomas Prime.

### 3.5 Agent Runtime

Runs specialized agents.

Responsibilities:

- Understand context.
- Generate plans and options.
- Select tools or programs.
- Interpret results.
- Produce structured Agent Outputs.

### 3.6 Program Runtime

Runs deterministic programs.

Responsibilities:

- Validate inputs.
- Execute repeatable logic.
- Return structured Program Results.
- Log execution, version, duration, and errors.
- Fail safely.

### 3.7 Validation Layer

Performs independent review.

Responsibilities:

- Check logic, evidence, assumptions, omissions, risk, and quality.
- Produce validation results without modifying the original output.
- Escalate uncertain or high-risk findings to Policy Engine or Thomas Prime.

### 3.8 Policy and Permission Engine

Controls risk and permission.

Responsibilities:

- Classify tool/action risk.
- Check agent, program, department, and user permissions.
- Decide whether an action is allowed, denied, reported, or sent to Thomas for approval.
- Prevent permission escalation by agents.
- Produce structured Permission Decisions.

### 3.9 Restricted Execution Service

Executes approved external actions.

Responsibilities:

- Execute only approved requests.
- Re-check inputs, permissions, and risk before execution.
- Record exact execution results.
- Run rollback or recovery procedures when available.

It must not:

- Create goals.
- Change strategy.
- Change risk policy.
- Expand permissions.
- Override approvals.

### 3.10 Memory System

Stores reusable knowledge and experience.

Memory scopes:

- Family Memory: Thomas identity, values, long-term goals, common decisions, common risk policy.
- Department Memory: domain knowledge, past work, judgments, performance, failures, lessons.

### 3.11 Audit Log

Records the system's behavior for later review.

Required coverage:

- Task creation.
- Routing.
- Agent decisions.
- Program executions.
- Validation results.
- Permission decisions.
- External executions.
- Memory writes and updates.

## 4. Standard Task Lifecycle

```text
1. Telegram message received
2. Task created
3. Thomas Prime interprets intent
4. Department Router selects department or direct program path
5. Department Thomas creates work plan
6. Agent Runtime and/or Program Runtime performs work
7. Validation Layer reviews output when required
8. Policy Engine checks risk and permission
9. Thomas approval is requested if required
10. Restricted Execution Service performs approved external action
11. Result is evaluated
12. Memory candidates are created
13. Audit Log records the flow
14. Telegram Gateway responds to Thomas
```

## 5. Contract Conventions

All core objects must follow these conventions.

### 5.1 Common Fields

Every major record should include:

| Field | Meaning |
| --- | --- |
| `schema_version` | Contract version, such as `task.v0.2` |
| `id` | Unique record ID |
| `trace_id` | End-to-end flow ID |
| `task_id` | Root or current Task ID |
| `parent_id` | Parent record ID when nested |
| `created_at` | UTC timestamp |
| `created_by` | Actor that created the record |
| `status` | Current state |
| `source` | Origin of the record |
| `audit_refs` | Audit event IDs related to the record |

### 5.2 Time

Storage timestamps use UTC. Telegram-facing summaries may display Korea time.

### 5.3 ID Prefixes

Recommended prefixes:

| Prefix | Object |
| --- | --- |
| `task_` | Task |
| `agentout_` | Agent Output |
| `programres_` | Program Result |
| `perm_` | Permission Decision |
| `mem_` | Memory Record |
| `audit_` | Audit Event |
| `exec_` | Execution Request or Result |

### 5.4 Risk Levels

| Level | Meaning |
| --- | --- |
| `none` | No meaningful risk |
| `low` | Reversible, private, or low-impact |
| `medium` | May affect existing data, schedule, content, or external-facing drafts |
| `high` | External publication, financial impact, operational changes, or sensitive data |
| `critical` | Irreversible, privileged, destructive, legal, trading, payment, or deployment-sensitive |

### 5.5 Permission Decisions

| Decision | Meaning |
| --- | --- |
| `allow` | Execute autonomously |
| `allow_with_report` | Execute and report afterward |
| `require_approval` | Ask Thomas before execution |
| `deny` | Block action |
| `escalate` | Send to Thomas Prime or higher review |

### 5.6 Autonomy Levels

| Level | Meaning |
| --- | --- |
| `L0_suggest_only` | Can only suggest |
| `L1_draft` | Can create draft outputs |
| `L2_internal_execute` | Can run internal, reversible actions |
| `L3_external_reversible` | Can perform limited reversible external actions |
| `L4_approval_required` | Requires Thomas approval |
| `L5_forbidden` | Not allowed |

## 6. Contract: Task

A Task is the universal work unit. Every request, scheduled job, agent-created subtask, or approval follow-up becomes a Task.

## 7. Contract: Agent Output

Agent Output records what an agent concluded and why. It must be structured enough for validation, policy checks, memory storage, and future learning.

### 7.1 Required Fields

| Field | Type | Meaning |
| --- | --- | --- |
| `schema_version` | string | `agent_output.v0.1` |
| `agent_output_id` | string | Unique output ID |
| `task_id` | string | Related Task |
| `trace_id` | string | End-to-end trace ID |
| `agent_id` | string | Agent that produced the output |
| `department_id` | string or null | Department context |
| `agent_role` | string | Research, Creation, Validation, Planning, etc. |
| `goal` | string | Goal interpreted by the agent |
| `current_situation` | string | Context summary |
| `used_information` | array | Source references and inputs used |
| `assumptions` | array | Key assumptions |
| `options_generated` | array | Options considered |
| `selected_option` | object or null | Chosen option |
| `decision_rationale` | string | Why this option was selected |
| `confidence` | number | 0.0 to 1.0 |
| `risk_assessment` | object | Risk estimate and reasons |
| `tool_requests` | array | Tools the agent wants to use or used |
| `program_requests` | array | Programs the agent wants to run or ran |
| `validation_required` | boolean | Whether independent validation is required |
| `permission_required` | boolean | Whether permission decision is required |
| `next_actions` | array | Proposed next steps |
| `memory_candidates` | array | Items that may become Memory Records |
| `response_to_thomas` | string | Human-facing summary |
| `limitations` | array | Known uncertainty or missing info |
| `status` | string | `draft`, `final`, `needs_validation`, or `rejected` |
| `created_at` | string | UTC timestamp |

### 7.2 Confidence Rule

Confidence alone does not decide autonomy. Autonomy is determined by:

```text
confidence x risk_level x permission_policy
```

Low confidence and low risk may trigger more research or a small experiment. Low confidence and high risk must stop execution or request Thomas approval.

## 9. Contract: Permission Decision

Permission Decision records whether a requested action may proceed.

### 9.1 Required Fields

| Field | Type | Meaning |
| --- | --- | --- |
| `schema_version` | string | `permission_decision.v0.2` |
| `permission_decision_id` | string | Unique decision ID |
| `task_id` | string | Related Task |
| `trace_id` | string | End-to-end trace ID |
| `requested_by` | object | Agent, Department, Program, or system component |
| `requested_action` | object | Exact action requested |
| `action_fingerprint` | string | Hash of normalized target, content, amount, Tool, and scope |
| `tool_class` | string | Read, Analysis, Creation, Modification, External Action, Destructive, Privileged |
| `action_scope` | string | Private, internal, external, financial, operational, etc. |
| `external_impact` | string | None, reversible, public, financial, operational, irreversible |
| `reversibility` | string | `reversible`, `partially_reversible`, or `irreversible` |
| `risk_level` | string | `GREEN`, `YELLOW`, `ORANGE`, or `RED` |
| `rules_evaluated` | array | Policy rules checked |
| `decision` | string | `ALLOW`, `EXECUTE_AND_REPORT`, `APPROVAL_REQUIRED`, or `BLOCK` |
| `decision_reasons` | array | Reasons for the decision |
| `required_approver` | string or null | Usually Thomas for high-risk actions |
| `approval_prompt` | string or null | Telegram message to request approval |
| `approval_id` | string or null | Unique approval request bound to this exact action |
| `one_time_use` | boolean | Whether an approval can be consumed only once |
| `constraints` | array | Conditions attached to approval |
| `expires_at` | string or null | Expiration timestamp |
| `created_at` | string | UTC timestamp |

### 9.2 Tool Class Default Policy

| Tool Class | Default Risk | Default Policy |
| --- | --- | --- |
| `read` | low | Mostly autonomous |
| `analysis` | low | Mostly autonomous |
| `creation` | low to medium | Autonomous or report afterward |
| `modification` | medium | Depends on scope |
| `external_action` | medium to high | Apply risk policy |
| `destructive` | high | Require approval or deny |
| `privileged` | critical | Thomas approval or forbidden |

### 9.3 Approval Binding Rule

When `decision` is `APPROVAL_REQUIRED`:

1. `approval_id`, `action_fingerprint`, exact target, action summary, expected cost, reversibility, and expiration must be present.
2. Any change to target, content, amount, Tool, or permission scope invalidates the approval request.
3. Thomas approval creates an immutable Audit Event containing approver ID, decision, action fingerprint, and decision time.
4. Restricted Execution Service consumes the approval once and records a separate execution event.
5. Duplicate execution with the same `approval_id` and `action_fingerprint` must be rejected.

| `updated_at` | string | UTC timestamp |
| `status` | string | `candidate`, `active`, `deprecated`, `rejected`, or `archived` |

### 10.2 Memory Write Rules

1. Family Memory updates require stronger validation than Department Memory.
2. Core identity, values, long-term goals, risk policy, permissions, and approval rules cannot be auto-changed.
3. In the MVP, every Agent-created memory starts as `candidate`.
4. Memory Records must preserve evidence references.
5. Contradictory memories must be versioned, not silently overwritten.

## 11. Supporting Contracts To Define Next

The five contracts above are the foundation. The next contracts should be:

1. `Tool Request`: how Agents request tools.
2. `Execution Request`: how approved external actions are handed to Restricted Execution Service.
3. `Execution Result`: how external action results are recorded.
4. `Validation Result`: how independent review is represented.
5. `Audit Event`: how every meaningful step is logged.
6. `Department Definition`: how departments are registered without hard-coding them.
7. `Program Definition`: how deterministic programs are registered and versioned.

Implemented companion contracts:

- [MVP Dynamic Role Contract](../03_ROLE_CONTRACTS/MVP_DYNAMIC_ROLE_CONTRACT.md): how Dynamic Roles are limited and evaluated.
- [Role Assignment Contract](../03_ROLE_CONTRACTS/ROLE_ASSIGNMENT_CONTRACT.md): how a Role receives Task-specific Context, permission, tools, and budget.

## 12. Initial Non-Implementation Decisions

These decisions are intentionally fixed before code:

1. Telegram is the first user interface.
2. Server runtime is assumed.
3. Department structure remains flexible.
4. All work flows through Task.
5. Agent outputs must be structured.
6. Program results must be deterministic and auditable where possible.
7. Permission decisions are first-class records.
8. Memory writes are explicit records, not hidden side effects.
9. External actions require Restricted Execution Service.
10. Core Thomas identity, values, goals, risk policy, and permissions cannot be self-modified.

## 13. Design Decisions and Open Questions

Resolved for the MVP:

1. Telegram uses one authenticated Thomas private chat as the Control Channel. Groups, channels, and other users are not autonomous targets.
2. Control Channel responses, reports, and approval requests are autonomous. Third-party or public communications require action-bound approval.
3. Agent-created memory starts as a candidate. Prime may promote low-risk operational memory; Thomas alone approves Core promotion.

Still open before implementation:

1. Should Telegram messages be stored verbatim, summarized, or selectively stored?
2. What is the first memory backend: file, SQLite, Postgres, or vector database plus relational store?
3. What is the first queue model: simple database queue, Redis, or external job queue?
4. Should Thomas Prime be a single agent, a deterministic coordinator plus agent, or a hybrid?
5. What is the minimum dashboard or admin view beyond Telegram?
6. What numeric token and cost budgets should the first server runtime use?
